# 阶段E演示讲稿（5-8分钟）

## 0. 开场（30秒）
- 今天演示的是“金融领域董秘回答判别系统”的阶段E版本，重点是把模型能力升级为可稳定运行的生产链路。
- 我会按“问题 -> 方案 -> 指标 -> 迭代”四段展示，并现场跑通关键流程。

## 1. 问题（1分钟）
- 业务问题：董秘回答存在“正面回应/逃避回应”混杂，人工筛查成本高，且需要识别财务实体与逃避战术。
- 工程问题：
  - 原有流程更偏开发态，编排层稳定性不足。
  - 异步任务与队列闭环缺少关键可靠性测试。
  - 演示素材分散，难以对外清晰讲述“价值与迭代”。

## 2. 方案（2分钟）
- 架构方案：阶段E统一为 `api + worker + redis + postgres` 四服务编排。
- 关键落地：
  - `api/worker` 复用单一后端镜像，移除运行时 `pip install`。
  - `api` 使用非 `--reload` 启动，`worker` 固定 Celery 参数。
  - `postgres/redis/api` 增加健康检查，依赖统一 `service_healthy`。
  - Compose 场景强制 `QA_DATABASE_FALLBACK_TO_SQLITE=false`，保证演示链路严格依赖 Postgres。
- 业务保持不变：不改现有业务 API 与响应结构，前后端兼容。

## 3. 指标（1.5分钟）
- 模型指标来源：`notebooks/evaluation/student_compare/model_summary.csv`。
- 工程指标（阶段E新增口径）：
  - 启动链路：四服务健康后 API 可用。
  - 队列闭环：低置信度样本可入队，手动入队可兜底。
  - 异步可靠性：Agent 建议任务成功/失败可观测，可轮询状态。
  - 失败暴露：关闭 fallback 后数据库故障可 fail-fast。

## 4. 迭代（1分钟）
- 数据飞轮：低置信度与人工复核结果回流训练集，持续提升子标签质量。
- 后续方向：
  - 增量微调与自动评估流水线。
  - 更细粒度的审计报表与多角色权限。
  - 观测能力（任务SLA、失败分类、恢复时长）进一步产品化。

## 5. 现场演示流程（2分钟）
1. 启动编排：
```bash
docker compose up --build -d postgres redis api worker
docker compose ps
```
2. 健康检查：
```bash
curl http://127.0.0.1:8000/api/health
```
3. 前端演示主链路：
- 分析页输入问答对 -> 获取判别结果。
- 手动将样本加入待复核队列。
- 在复核页触发 Agent 建议并轮询任务状态。
- 展示复核结果与导出能力。
4. 故障演示（可选）：
- 关闭数据库连接或配置错误，验证 fallback 关闭时系统快速报错，不静默回退。

## 6. 收尾（30秒）
- 阶段E的价值不在“多加了几个服务”，而在“把模型能力转为可验证、可恢复、可演示的工程系统”。
- 这为后续真实场景部署和持续迭代打下了可复用基础。
